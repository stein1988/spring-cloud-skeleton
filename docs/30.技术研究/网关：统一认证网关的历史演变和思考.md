## 3、详细讲讲网关的历史演变

**核心洞察**：网关是分布式系统的门面（Facade）模式在网络层的投影。它的演进史，本质上是关注点分离原则在架构垂直方向上的持续下沉——从应用代码中剥离，到基础设施层沉淀，最终进化为云原生时代的流量操作系统。

---

### 一、史前时代：单体蛮荒与直连之痛（2000-2010）

在微服务概念诞生之前，"网关"并不存在独立的架构地位。那时的 Nginx 只是反向代理（Reverse Proxy），职责单一：隐藏 Tomcat 的 8080 端口，提供静态文件缓存。

**架构病症：客户端直连的耦合噩梦**

```
┌─────────────┐         ┌─────────────────────────────┐
│   浏览器     │ ──────→ │  Nginx (静态资源)            │
└─────────────┘         └─────────────────────────────┘
                               ↓
                       ┌─────────────────────────────┐
                       │   Tomcat (单体应用)          │
                       │   - 鉴权代码散落在 Servlet    │
                       │   - 跨域配置写在 web.xml      │
                       │   - 限流逻辑硬编码在 Interceptor│
                       └─────────────────────────────┘
```

**致命缺陷**：

1. **客户端臃肿**：移动端需要对接 10+ 个后端服务，处理 10 套鉴权逻辑
2. **协议僵化**：后端想从 HTTP/1.1 升级到 HTTP/2，需要所有客户端同步升级
3. **雪崩效应**：单个服务超时导致客户端线程池耗尽，级联故障

**架构师觉醒**：必须有一个统一的前置层，作为系统的 **Facade**，对外暴露统一接口，对内隔离复杂度。这催生了第一代 API 网关。

---

### 二、第一代网关：边缘路由与 Servlet 桎梏（2010-2016）

Netflix 开源的 Zuul 1.x 定义了第一代微服务网关的标准模型。它基于 Servlet 构建，采用阻塞式 IO（Blocking IO）线程模型。

**架构范式：网关作为"安检大门"**

```
┌─────────────┐     ┌──────────────────────────────┐     ┌─────────────────┐
│   客户端     │─────→│  Zuul 1.x (Servlet 容器)     │─────→│  微服务集群      │
└─────────────┘     │  - 路由映射 (Ribbon)          │     └─────────────────┘
                    │  - 过滤器链 (Filter Chain)    │
                    │  - 同步阻塞调用后端            │
                    └──────────────────────────────┘
```

**核心贡献**：

- **前置鉴权**：JWT 验证从业务服务抽离，统一在网关层拦截
- **路由抽象**：`/api/users/**` → 用户服务，`/api/orders/**` → 订单服务
- **客户端简化**：客户端只需记住网关域名，无需关心服务发现

**性能天花板**：Zuul 1.x 采用 One Thread Per Connection 模型。当后端服务响应慢时，线程池迅速耗尽。在 Netflix 的生产环境中，这导致了 C10K 问题的复现——连接数超过 10,000 时，系统吞吐量断崖式下跌。

**架构师困境**：网关本应是轻量级入口，却因 Servlet 的阻塞模型变成了性能瓶颈。这倒逼了第二代网关的技术选型革命。

---

### 三、第二代网关：反应式革命与 Spring Cloud Gateway（2016-2020）

2016 年，Spring 团队推出了 Spring Cloud Gateway（以下简称 SCG），这不仅是工具的更迭，更是编程模型的范式转移——从 Servlet 的命令式编程转向 WebFlux 的反应式编程（Reactive Programming）。

**架构跃迁：Netty 与事件循环**

```java
// 传统 Zuul 1.x：阻塞式，线程等待
public Object route(Request req) {
    Response resp = httpClient.call(req); // 线程阻塞等待，啥也干不了
    return transform(resp);
}

// SCG：非阻塞式，事件驱动
public Mono<ServerResponse> route(ServerRequest req) {
    return webClient.get()
        .uri("lb://user-service/users")
        .retrieve()
        .bodyToMono(String.class)  // 立即返回，不阻塞线程
        .map(this::transform);
}
```

**技术选型对比**：

| 特性 | Zuul 1.x | Spring Cloud Gateway |
|------|----------|---------------------|
| IO模型 | 阻塞IO | 非阻塞IO (Netty) |
| 线程模型 | One Thread Per Connection | Event Loop |
| 吞吐量 | 低 | 高 |
| 背压处理 | 无 | 内置响应式背压 |
| 编程模型 | 命令式 | 声明式响应式 |

**SCG 的Filter机制**：

```java
@Component
public class AuthFilter implements GlobalFilter, Ordered {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        
        if (!isValid(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        
        return chain.filter(exchange);
    }
    
    @Override
    public int getOrder() {
        return -100; // 优先级最高
    }
}
```

### 四、第三代网关：云原生与 Service Mesh（2020-至今）

**架构演进：从网关到服务网格**

```
┌─────────────────────────────────────────────────────────┐
│                      控制平面 (Control Plane)            │
│                   Istio / Apricot Control              │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                      数据平面 (Data Plane)              │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐   │
│  │ Sidecar │  │ Sidecar │  │ Sidecar │  │ Sidecar │   │
│  │ Proxy A │  │ Proxy B │  │ Proxy C │  │ Proxy D │   │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘   │
└─────────────────────────────────────────────────────────┘
```

**核心变革**：

1. **Sidecar 模式**：每个服务实例旁部署一个代理（Envoy/Mosn），实现流量拦截
2. **关注点分离**：网关专注入口流量，服务间流量由 Sidecar 处理
3. **零信任安全**：服务间通信也需认证，不再依赖网络隔离

**云原生网关的典型代表**：

| 特性 | Envoy | APISIX | Higress |
|------|-------|--------|---------|
| 架构 | Sidecar | 可作为网关或Sidecar | 云原生Ingress |
| 协议支持 | HTTP/gRPC/TCP | HTTP/gRPC/TCP/Dubbo | HTTP/gRPC/Dubbo |
| 生态集成 | Istio | APISIX Dashboard | 阿里云/容器服务 |
| 性能 | 高 | 极高 | 高 |

### 五、认证鉴权架构演进

**架构演进图**：

```
┌──────────────────────────────────────────────────────────────────┐
│                      演进阶段对比                                  │
├──────────────────────────────────────────────────────────────────┤
│                                                                   │
│  阶段1: 应用内鉴权 (2010前)                                        │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐                     │
│  │ App     │────→│ Service │────→│ 验证JWT  │                     │
│  │         │     │   A     │     │ 复制4份  │                     │
│  └─────────┘     └─────────┘     └─────────┘                     │
│                                                                   │
│  阶段2: 网关统一鉴权 (2010-2020)                                   │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐                     │
│  │ App     │────→│ Gateway │────→│ Service │                     │
│  │         │     │ 验JWT   │     │  免鉴权  │                     │
│  └─────────┘     └─────────┘     └─────────┘                     │
│                                                                   │
│  阶段3: 零信任鉴权 (2020后)                                        │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐                     │
│  │ App     │────→│ Sidecar │────→│ Service │                     │
│  │         │     │ mTLS    │     │ 验Token │                     │
│  └─────────┘     └─────────┘     └─────────┘                     │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

**鉴权方式对比**：

| 鉴权方式 | 适用场景 | 优点 | 缺点 |
|---------|---------|------|------|
| **Session/Cookie** | 传统Web应用 | 实现简单 | 无法跨域，扩展性差 |
| **JWT** | 微服务API | 无状态，易扩展 | 注销困难，存储受限 |
| **OAuth2 + JWT** | 第三方登录 | 支持授权协议 | 实现复杂 |
| **mTLS** | 服务间通信 | 双向认证，安全性高 | 证书管理复杂 |

### 六、网关选型建议

**技术选型矩阵**：

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| **中小企业，HTTP服务为主** | Spring Cloud Gateway | 生态完善，学习成本低 |
| **大规模高性能需求** | APISIX / Envoy | 高性能，云原生友好 |
| **阿里云生态** | Higress / MSE | 深度集成，开箱即用 |
| **多语言微服务** | Istio + Envoy | 统一流量管理 |
| **遗留系统改造** | Kong / Zuul 2.x | 插件丰富，兼容性好 |

**决策树**：

```
开始选型
    ↓
是否需要云原生集成？ ──否──→ 是否需要高性能？ ──否──→ Spring Cloud Gateway
    ↓是                                  ↓是
是否在阿里云？                         APISIX
    ↓           ↓                  
   是          否                   
    ↓           ↓                  
  Higress    Istio + Envoy
```

### 总结

网关的演进史，本质上是**关注点分离**原则在架构垂直方向上的持续下沉：

1. **第一代**：从应用代码中抽离鉴权、路由到独立网关
2. **第二代**：从阻塞IO到非阻塞IO，提升性能
3. **第三代**：从集中式网关到分布式 Sidecar，实现零信任

理解网关的演进逻辑，才能在技术选型时做出正确决策。

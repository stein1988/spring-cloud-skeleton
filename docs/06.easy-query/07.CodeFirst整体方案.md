# CodeFirst 整体方案

## 1. CodeFirst 概念介绍

CodeFirst 是一种特殊的开发方式，不同于常规的 DB-First（先开发数据库后开发代码），CodeFirst 是先开发业务逻辑、先写代码，后续通过代码生成数据库所需的表结构。

### 1.1 核心特性

- **自动表结构同步**：根据实体类自动创建表、添加列、修改列名等
- **类型自动映射**：将 Java 类型自动映射到对应的数据库类型
- **注释自动生成**：将实体类和属性的注释自动生成为数据库表和列的注释
- **事务支持**：在执行表结构同步时支持事务处理

### 1.2 适用场景

- **开发环境**：快速原型开发，频繁的表结构变更
- **新项目初始化**：从零开始的项目，先设计代码结构再生成数据库
- **测试环境**：自动化测试时的数据库准备

## 2. SystemService 的封装与 Controller 开放接口

### 2.1 SystemService 核心实现

`SystemService` 是对 Easy-Query CodeFirst 功能的封装，提供了以下核心功能：

- **实体类扫描**：从指定包路径自动扫描带有 `@Table` 注解的实体类
- **表结构同步**：执行表结构的创建和更新
- **SQL 日志输出**：打印执行的 SQL 语句，便于调试

关键代码实现：

```java
public SystemService(EasyEntityQuery easyEntityQuery, String... packagePaths) {
    this.easyEntityQuery = easyEntityQuery;
    this.entities = scanEntityProxyClasses(packagePaths);
}

private List<Class<?>> scanEntityProxyClasses(String... packagePaths) {
    ClassPathScanningCandidateComponentProvider provider = new ClassPathScanningCandidateComponentProvider(false);
    provider.addIncludeFilter(new AnnotationTypeFilter(Table.class));

    List<Class<?>> classes = new ArrayList<>();
    for (String packagePath : packagePaths) {
        Set<BeanDefinition> components = provider.findCandidateComponents(packagePath);
        for (BeanDefinition component : components) {
            try {
                classes.add(ClassUtils.forName(Objects.requireNonNull(component.getBeanClassName()), null));
            } catch (ClassNotFoundException e) {
                log.error("Failed to load class: {}", component.getBeanClassName(), e);
            }
        }
    }
    classes.forEach(clazz -> log.info("Scanned entity proxy class: {}", clazz.getName()));
    return classes;
}

public void syncTable() {
    easyEntityQuery.setMigrationParser(new MigrationEntityParser());
    DatabaseCodeFirst databaseCodeFirst = easyEntityQuery.getDatabaseCodeFirst();
    CodeFirstCommand codeFirstCommand = databaseCodeFirst.syncTableCommand(entities);
    codeFirstCommand.executeWithTransaction(arg -> {
        log.info(arg.getSQL());
        arg.commit();
    });
}
```

### 2.2 实现类 SystemServiceImpl

`SystemServiceImpl` 继承自 `SystemService`，指定了实体类的扫描路径：

```java
@Service
public class SystemServiceImpl extends SystemService {
    public SystemServiceImpl(EasyEntityQuery easyEntityQuery) {
        super(easyEntityQuery, "com.lonbon.cloud.user.domain.entity");
    }
}
```

### 2.3 Controller 开放接口

通过 `SystemController` 暴露 HTTP 接口，方便前端或其他服务调用：

```java
@RestController
@RequestMapping("/system")
@Tag(name = "系统", description = "系统级操作，谨慎使用")
public class SystemController {

    @Resource
    private SystemService systemService;

    @PostMapping("/sync-table")
    @Operation(summary = "同步表", description = "code first 同步表")
    public Response<Void> syncTable() {
        systemService.syncTable();
        return Response.success();
    }
}
```

## 3. 实体类的自动导入扫描策略

### 3.1 扫描机制

- **基于 Spring 的 ClassPathScanningCandidateComponentProvider**：自动扫描指定包路径下的类
- **注解过滤**：只扫描带有 `@Table` 注解的类
- **类加载**：通过反射加载扫描到的类

### 3.2 扫描路径配置

在 `SystemServiceImpl` 中配置扫描路径，支持多个包路径：

```java
// 单个包路径
super(easyEntityQuery, "com.lonbon.cloud.user.domain.entity");

// 多个包路径（示例）
super(easyEntityQuery, 
    "com.lonbon.cloud.user.domain.entity",
    "com.lonbon.cloud.demo.domain.entity"
);
```

## 4. 基于 PostgreSQL 的类型定制处理

### 4.1 类型映射配置

通过 `MigrationEntityParser` 类定制 PostgreSQL 的类型映射：

```java
private static final Map<Class<?>, ColumnDbTypeResult> columnTypeMap = new HashMap<>();

static {
    columnTypeMap.put(boolean.class, new ColumnDbTypeResult("BOOL", "FALSE")); // boolean默认为 NOT NULL DEFAULT FALSE
    columnTypeMap.put(int.class, new ColumnDbTypeResult("INT4", "0"));
    columnTypeMap.put(String.class, new ColumnDbTypeResult("TEXT", "''"));   // String默认为NOT NULL DEFAULT "";
    columnTypeMap.put(UUID.class, new ColumnDbTypeResult("UUID", null));
    columnTypeMap.put(OffsetDateTime.class, new ColumnDbTypeResult("TIMESTAMPTZ", null));
    columnTypeMap.put(LocalDate.class, new ColumnDbTypeResult("DATE", null));
}
```

### 4.2 允许类型列表

| Java 类型 | PostgreSQL 类型 | 默认值 | 说明 |
|----------|----------------|--------|------|
| `boolean` | `BOOL` | `FALSE` | 基本类型默认为 NOT NULL DEFAULT FALSE |
| `int` | `INT4` | `0` | 基本类型默认为 NOT NULL DEFAULT 0 |
| `UUID` | `UUID` | `null` | 用于主键和关联字段 |
| `String` | `TEXT` | `''` | 默认 NOT NULL DEFAULT "" |
| `OffsetDateTime` | `TIMESTAMPTZ` | `null` | 带时区的时间戳 |
| `LocalDate` | `DATE` | `null` | 用于存储记录型的日期 |

### 4.3 Column 注解的写法规范

#### 4.3.1 主键字段

```java
/**
 * 主键ID，使用UUIDv7算法
 */
@Column(comment = "主键ID", primaryKey = true, primaryKeyGenerator = UUIDPrimaryKeyGenerator.class)
protected UUID id;
```

#### 4.3.2 普通字段

```java
/**
 * 租户ID
 */
@Column(comment = "租户ID")
protected UUID tenantId;
```

#### 4.3.3 带默认值的字段

```java
/**
 * 创建时间，UTC时间戳
 */
@Column(comment = "创建时间", nullable = false, dbDefault = "NOW()")
@UpdateIgnore
protected OffsetDateTime createTime;
```

#### 4.3.4 逻辑删除字段

```java
/**
 * 逻辑删除标志：是否删除，默认值为false
 */
@Column(comment = "是否删除")
@LogicDelete(strategy = LogicDeleteStrategyEnum.CUSTOM, strategyName = "DEFAULT_LOGIC_DELETE_STRATEGY")
@UpdateIgnore
protected boolean isDelete;
```

### 5.1 自定义解析器配置

通过 `MyConfiguration` 和 `MyStarterConfigurer` 配置自定义的数据库迁移提供者：

```java
@Configuration
public class MyConfiguration {
    @Bean("MyStarterConfigurer")
    @Primary
    public StarterConfigurer starterConfigurer() {
        return new MyStarterConfigurer();
    }
}

public class MyStarterConfigurer implements StarterConfigurer {
    @Override
    public void configure(ServiceCollection services) {
        services.addService(DatabaseMigrationProvider.class, MyDatabaseMigrationProvider.class);
    }
}
```

## 6. 当前现状与未解决的问题

### 6.1 当前适用场景

- **开发环境**：适合开发环境的数据表创建和结构变更
- **数据库脚本自动生成**：通过日志输出可以获取执行的 SQL 脚本

### 6.2 未解决的问题

#### 6.2.1 缺少数据库版本号功能

- **问题**：当前实现没有数据库版本管理机制
- **影响**：无法追踪数据库结构变更历史，难以回滚到特定版本
- **建议**：集成数据库版本管理工具，如 Flyway 或 Liquibase

#### 6.2.2 缺少预设数据同步功能

- **问题**：只能同步表结构，无法同步初始数据
- **影响**：系统初始化时需要手动导入基础数据
- **建议**：添加预设数据同步机制，支持从配置文件或代码中加载初始数据

#### 6.2.3 其他数据类型的适配

- **问题**：对一些特殊数据类型的支持不够完善
- **影响**：复杂数据类型可能无法正确映射到数据库
- **建议**：扩展类型映射，支持更多数据类型，如枚举、JSON 等

#### 6.2.4 生产环境安全性

- **问题**：当前实现直接执行 SQL 语句，没有权限控制和审批机制
- **影响**：在生产环境使用可能存在安全风险
- **建议**：添加环境检测，生产环境需要额外的审批流程

## 7. 使用建议

1. **开发环境**：可以直接使用 `/system/sync-table` 接口同步表结构
2. **测试环境**：建议在部署前执行同步操作
3. **生产环境**：
   - 建议先在测试环境验证 SQL 语句
   - 手动执行经过审核的 SQL 脚本
   - 定期备份数据库

4. **最佳实践**：
   - 为实体类添加详细的注释
   - 合理使用 `@Column` 注解的属性
   - 遵循统一的命名规范
   - 定期清理未使用的实体类

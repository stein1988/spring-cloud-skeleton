# easy-query 框架自定义配置

## 1. 概述

本文档介绍了在 easy-query 框架下如何实现公共字段的自动管理，包括 BaseEntity 设计、拦截器配置、逻辑删除策略等内容。通过这些配置，可以实现公共字段的自动填充和管理，减少重复代码，提高开发效率。

## 2. BaseEntity 设计

### 2.1 实体基类定义

实体基类 `BaseEntity` 定义了所有实体类的公共字段，包括主键、逻辑删除标志、时间戳字段和版本号字段。详细代码见：`com.lonbon.cloud.base.entity.BaseEntity`

### 2.2 注解说明

- **@Column**：标记字段为主键，并指定主键生成器
- **@LogicDelete**：标记字段为逻辑删除字段，指定自定义删除策略
- **@UpdateIgnore**：标记字段在更新操作时忽略，避免被自动更新覆盖
- **@Version**：标记字段为版本号字段，用于乐观锁

**参考链接**：

- [easy-query 官方文档|实战|设计|对象设计](https://www.easy-query.com/easy-query-doc/practice/configuration/entity.html)

## 3. 拦截器与策略配置

### 3.1 DefaultEntityInterceptor

`DefaultEntityInterceptor` 实现了 easy-query 的拦截器接口，用于自动处理实体的创建和更新时间。详细代码见：`com.lonbon.cloud.base.entity.DefaultEntityInterceptor`

**功能说明**：

- **自动应用范围**：所有继承 `BaseEntity` 的实体类
- **insert 操作**：自动设置 `createdAt` 和 `updatedAt` 为当前时间（如果为空）
- **update 操作**：自动更新 `updatedAt` 为当前时间
- **扩展点**：预留了设置 `createdBy` 和 `updatedBy` 的位置，可以集成用户认证系统获取当前用户ID

**参考链接**：

- [easy-query 官方文档|高级功能|全局拦截器](https://www.easy-query.com/easy-query-doc/adv/interceptor.html)

### 3.2 DefaultLogicDeleteStrategy

`DefaultLogicDeleteStrategy` 实现了 easy-query 的逻辑删除策略接口，用于处理逻辑删除相关的字段。详细代码见：`com.lonbon.cloud.base.entity.DefaultLogicDeleteStrategy`

**功能说明**：

- **策略名称**：`DEFAULT_LOGIC_DELETE_STRATEGY`，与 `BaseEntity` 中 `@LogicDelete` 注解的 `strategyName` 对应
- **查询过滤器**：自动为查询操作添加 `isDeleted = false` 的条件，确保只查询未删除的记录
- **逻辑删除操作**：执行逻辑删除时，自动设置 `isDeleted = true` 和 `deletedAt` 为当前时间
- **扩展点**：预留了设置 `deletedBy` 的位置，可以集成用户认证系统获取当前用户ID

**参考链接**：

- [easy-query 官方文档|高级功能|逻辑删除](https://www.easy-query.com/easy-query-doc/adv/logic-delete.html)

### 3.3 UUIDPrimaryKeyGenerator

`UUIDPrimaryKeyGenerator` 实现了 easy-query 的主键生成器接口，用于生成 UUIDv7 类型的主键。详细代码见：`com.lonbon.cloud.base.entity.UUIDPrimaryKeyGenerator`

**功能说明**：

- **主键类型**：UUIDv7 类型，全局唯一
- **生成策略**：基于当前时间戳和随机数生成 UUIDv7 字符串

**参考链接**：

- [easy-query 官方文档|高级功能|自定义主键](https://www.easy-query.com/easy-query-doc/adv/auto-key.html)

### 3.4 JdbcTypeHandlerConfigurer

`JdbcTypeHandlerConfigurer` 是TypeHandler配置接口。详细代码见：`com.lonbon.cloud.base.entity.JdbcTypeHandlerConfigurer`

**功能说明**：

- **自动注册**：实现 `JdbcTypeHandlerConfigurer` 接口，框架会自动扫描并注册实现类中的类型处理器
- **自定义类型处理器**：可以注册自定义的 JDBC 类型处理器，用于处理特定类型的字段

**参考链接**：

- [easy-query 官方文档|高级功能|自定义TypeHandler](https://www.easy-query.com/easy-query-doc/adv/type-handler.html)

#### 3.4.1 OffsetDateTimeTypeHandler

`OffsetDateTimeTypeHandler` 实现了 easy-query 的类型处理器接口，用于处理 `OffsetDateTime` 类型的字段。详细代码见：`com.lonbon.cloud.base.entity.OffsetDateTimeTypeHandler`

**功能说明**：

- **数据库类型**：`TIMESTAMP WITH TIME ZONE`
- **Java 类型**：`OffsetDateTime`
- **序列化**：将 `OffsetDateTime` 转换为数据库存储格式（`yyyy-MM-dd HH:mm:ss.SSSXXX`）
- **反序列化**：将数据库存储格式转换为 `OffsetDateTime` 对象

### 3.4.2 JSON 字段支持

#### 3.4.2.1 JsonObject 接口

`JsonObject` 是一个标记接口，用于标识需要进行 JSON 序列化和反序列化的对象。详细代码见：`com.lonbon.cloud.base.entity.JsonObject`

**功能说明**：

- **标记接口**：无任何方法，仅作为类型标记
- **应用场景**：所有需要存储为 JSON 格式的对象都应实现此接口

#### 3.4.2.2 JsonObjectAutoConverter

`JsonObjectAutoConverter` 实现了 easy-query 的 `ValueAutoConverter` 接口，用于自动处理 JSON 对象的序列化和反序列化。详细代码见：`com.lonbon.cloud.base.entity.JsonObjectAutoConverter`

**功能说明**：

- **自动应用范围**：所有实现 `JsonObject` 接口的类型
- **序列化**：将 Java 对象转换为 JSON 字符串
- **反序列化**：将 JSON 字符串转换为 Java 对象
- **支持类型**：支持 PostgreSQL 的 `jsonb` 类型（通过 `PGobject` 处理）

**实现原理**：

- 使用 `ONode` 库进行 JSON 序列化和反序列化
- 缓存字段类型信息，提高性能
- 支持多种输入类型（String、PGobject 等）

#### 3.4.2.3 JsonObjectTypeHandler

`JsonObjectTypeHandler` 实现了 `JdbcTypeHandlerConfigurer` 接口，用于处理 JSON 类型字段的 JDBC 操作。详细代码见：`com.lonbon.cloud.base.entity.JsonObjectTypeHandler`

**功能说明**：

- **处理类型**：`String` 类型（存储 JSON 字符串）
- **特殊处理**：当字段类型实现 `JsonObject` 接口且 JDBC 类型为 `OTHER` 时，使用 `PGobject` 处理 PostgreSQL 的 `jsonb` 类型
- **兼容性**：同时支持普通字符串和 JSON 字符串的处理

### 3.5 version

**工作原理**:

乐观锁是一种并发控制机制，通过版本号来检测并发更新冲突。其工作原理如下：

1. 当读取实体时，同时读取其版本号
2. 当更新实体时，将版本号作为 WHERE 条件的一部分
3. 更新操作同时将版本号递增
4. 如果在读取和更新之间，有其他线程修改了该实体，版本号会发生变化，导致更新操作失败

**配置与实现**:

在 `BaseEntity` 中，使用 `@Version` 注解标记版本号字段，并指定 `VersionIntStrategy` 策略实现版本号自动递增。

**乐观锁冲突处理**:

当发生乐观锁冲突时，通常的处理策略有：

1. **重试机制**：捕获异常后重新获取最新数据，重新尝试更新
2. **提示用户**：告知用户数据已被其他用户修改，请重新操作
3. **自动合并**：尝试自动合并用户的修改和其他用户的修改

**注意事项**:

- **版本号类型**：使用 int 类型的版本号适用于大多数场景，对于更新非常频繁的实体，可以考虑使用 long 类型
- **表达式更新**：使用表达式更新时，需要显式指定 `withVersion`，否则不会使用乐观锁
- **性能考虑**：乐观锁会增加数据库操作的复杂度，但避免了悲观锁的性能开销，适合并发冲突较少的场景
- **批量操作**：批量更新操作中，乐观锁的使用需要特别注意，确保版本号的正确处理

**参考链接**：

- [easy-query 官方文档|高级功能|乐观锁版本号](https://www.easy-query.com/easy-query-doc/adv/version.html)

## 4. Easy-Query 框架配置

### 4.1 EasyQueryConfiguration

`EasyQueryConfiguration` 用于注册和配置 Easy-Query 框架的各种组件，包括拦截器、逻辑删除策略、分片初始化、列加密、版本策略等。详细代码见：`com.lonbon.cloud.base.config.EasyQueryConfiguration`

**主要功能**：

- 注册各种扩展组件到 Easy-Query 框架中
- 配置 JDBC 类型处理器
- 注册路由策略

### 4.2 BaseSolonPlugin

`BaseSolonPlugin` 实现了 Solon 的 Plugin 接口，用于在应用启动时自动注册基础模块的各种公共组件。详细代码见：`com.lonbon.cloud.base.plugin.BaseSolonPlugin`

**注册的组件**：

- 序列化配置组件
- 全局异常过滤器
- UUID 类型转换器
- Easy-Query 相关组件（实体拦截器、逻辑删除策略、类型处理器、主键生成器、框架配置）

## 5. 配置与依赖

### 5.1 依赖配置

项目使用了以下关键依赖：

- **easy-query**：ORM框架，提供了拦截器和逻辑删除功能
- **uuid-creator**：用于生成UUIDv7
- **Solon**：提供了依赖注入和组件管理

### 5.2 组件注册

所有拦截器和策略类都使用了 `@Component` 注解，确保被 Solon 自动扫描和注册：

- `UUIDPrimaryKeyGenerator`：主键生成器
- `DefaultEntityInterceptor`：实体拦截器
- `DefaultLogicDeleteStrategy`：逻辑删除策略
- `OffsetDateTimeTypeHandler`：日期类型处理器
- `JsonObjectAutoConverter`：JSON对象自动转换器
- `JsonObjectTypeHandler`：JSON类型处理器

## 6. 最佳实践

1. **统一继承 BaseEntity**：所有实体类都应继承 `BaseEntity`，确保字段结构一致
2. **避免手动设置公共字段**：公共字段由拦截器自动处理，避免手动设置导致的不一致
3. **集成用户认证系统**：根据实际项目情况，集成用户认证系统，实现 `createdBy`、`updatedBy` 和 `deletedBy` 的自动设置
4. **合理使用版本号**：利用 `version` 字段实现乐观锁，防止并发更新冲突
5. **注意时区处理**：所有时间字段使用 `OffsetDateTime` 类型，确保跨时区一致性
6. **使用 JSON 字段**：对于复杂结构的字段，实现 `JsonObject` 接口，自动支持 JSON 序列化和反序列化
7. **数据库兼容性**：PostgreSQL 环境下，JSON 字段会自动使用 `jsonb` 类型存储，提高查询性能

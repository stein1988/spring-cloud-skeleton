# MapStruct Plus 基础用法

## 概述

MapStruct Plus 是一个高性能的对象转换框架，基于 MapStruct 进行了增强，提供了更简洁的 API 和更强大的功能。本项目已集成 MapStruct Plus 1.5.0 版本。

## 依赖配置

### Maven 依赖

在父 pom.xml 中已统一管理依赖，子模块只需引入 starter 即可：

```xml
<dependency>
    <groupId>io.github.linpeilie</groupId>
    <artifactId>mapstruct-plus-spring-boot-starter</artifactId>
    <version>${mapstruct-plus.version}</version>
</dependency>
```

### 编译配置

项目已配置好 Maven 编译器插件的 annotation processor：

```xml
<annotationProcessorPaths>
    <path>
        <groupId>org.projectlombok</groupId>
        <artifactId>lombok</artifactId>
    </path>
    <path>
        <groupId>com.easy-query</groupId>
        <artifactId>sql-processor</artifactId>
        <version>${easy-query.version}</version>
    </path>
    <path>
        <groupId>io.github.linpeilie</groupId>
        <artifactId>mapstruct-plus-processor</artifactId>
        <version>${mapstruct-plus.version}</version>
    </path>
    <path>
        <groupId>org.projectlombok</groupId>
        <artifactId>lombok-mapstruct-binding</artifactId>
        <version>0.2.0</version>
    </path>
</annotationProcessorPaths>
```

**注意**：以上几个编译插件的顺序不能随意调整，`lombok-mapstruct-binding` 用于解决 Lombok 与 MapStruct 整合时的兼容性问题，必须放在 annotation processor 路径的最后一位，`easy-query.sql-processor` 要在 `mapstruct-plus-processor` 的前面，否则数据库关联的实体类生成转换代码会有问题，原因不明。

## 策略配置

策略配置可在 pom.xml 的 `compilerArgs` 中修改：

### 已修改策略

#### nullValuePropertyMappingStrategy

- **说明**：属性值为 null 时的处理策略
- **类型**：`NullValuePropertyMappingStrategy`
- **可选项**：
  - `SET_TO_NULL`：设置为 null
  - `SET_TO_DEFAULT`：设置为默认值
  - `IGNORE`：忽略
- **默认值**：`SET_TO_NULL`
- **编译参数**：`-Amapstruct.plus.nullValuePropertyMappingStrategy`
- **修改值**：`IGNORE`
- **详细说明**：参考 [02.NULL属性的意义与处理策略.md](./02.NULL属性的意义与处理策略.md)

## 基本用法

### 1. 定义实体类

假设有两个类：业务实体 `User` 和数据传输对象 `UserDto`：

```java
@Data
public class User {
    private String username;
    private int age;
    private boolean young;
}

@Data
public class UserDto {
    private String username;
    private int age;
    private boolean young;
}
```

### 2. 指定对象映射关系

在需要转换的类上添加 `@AutoMapper` 注解，指定目标类型：

```java
@Data
@AutoMapper(target = UserDto.class)
public class User {
    private String username;
    private int age;
    private boolean young;
}
```

或者在 DTO 类上添加注解：

```java
@Data
@AutoMapper(target = User.class)
public class UserDto {
    private String username;
    private int age;
    private boolean young;
}
```

### 3. 使用 Converter 进行转换

在 Spring Boot 环境中，直接注入 `Converter` 即可：

```java
@Autowired
private Converter converter;

public UserDto convertToDto(User user) {
    return converter.convert(user, UserDto.class);
}

public User convertToEntity(UserDto userDto) {
    return converter.convert(userDto, User.class);
}
```

### 4. 字段名称不一致时的映射

当源类和目标类的字段名称不一致时，使用 `@AutoMapping` 注解进行指定：

```java
@Data
@AutoMapper(target = UserDto.class)
public class User {
    private String userName;
    private int userAge;

    @AutoMapping(target = "username", source = "userName")
    @AutoMapping(target = "age", source = "userAge")
    public static class UserMapping {
    }
}
```

## 常见问题

### 1. 没有生成转换接口

如果编译后没有生成转换接口，请检查：
- 确认 `@AutoMapper` 注解已添加到类上
- 执行 `mvn clean compile` 重新编译
- 检查 `target/generated-sources` 目录下是否生成了转换接口
- 确认 Spring Boot 能够扫描到生成接口所在的包

### 2. "cannot find converter from xxx to xxx"

可能原因：
- 依赖未正确加载，尝试重新加载 Maven 项目
- 如果使用了 `spring-boot-devtools`，可能导致类加载器问题，建议移除该依赖
- 检查是否有多余的 MapStruct 依赖导致冲突

### 3. NoSuchMethodError

如果在多模块项目中出现 `NoSuchMethodError`，可能是每个模块都生成了相同的 adapter 类。解决方案是为每个模块配置不同的 `adapterPackage`：

```xml
<compilerArgs>
    <arg>-Amapstruct.plus.adapterPackage=com.example.module1.adapter</arg>
</compilerArgs>
```

### 4. 依赖下载问题

如果遇到依赖下载失败，建议使用腾讯 Maven 镜像源：

```xml
<repository>
    <id>tencent</id>
    <url>https://mirrors.cloud.tencent.com/repository/maven/</url>
</repository>
```

## 最佳实践

1. **单向映射**：建议只在一方添加 `@AutoMapper` 注解，避免重复生成
2. **字段命名**：保持源类和目标类字段命名一致，减少手动映射配置
4. **集合转换**：Converter 支持集合类型的自动转换
5. **嵌套对象**：支持嵌套对象的自动映射，无需额外配置

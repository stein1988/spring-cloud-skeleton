# NULL属性的意义与处理策略

在本项目的RESTful API数据交互及持久化流程中，NULL属性的含义具有场景依赖性，其处理策略直接影响数据安全性与业务逻辑正确性。本文结合DTO（数据传输对象）用于create（创建）和update（更新）两种核心场景，拆解RESTful HTTP请求→DTO→Entity（业务实体）→ORM（对象关系映射）→数据库的全链路，明确各环节中NULL属性的具体意义，论证MapStruct Plus框架中nullValuePropertyMappingStrategy设为IGNORE策略的合理性与必要性。

本系统中，DTO作为前端与后端的核心数据载体，承接RESTful HTTP请求参数，经MapStruct Plus转换为Entity后，由ORM框架执行数据库操作完成持久化。NULL属性在不同业务场景、不同数据流转环节中含义截然不同，若处理策略不当（如采用默认的SET\_TO\_NULL），会引发数据异常、业务逻辑偏差等问题，因此需明确NULL属性的意义并制定适配的处理策略。

# 一、核心前提说明

1\.  本系统DTO仅用于接收create、update操作的请求参数，不承担查询返回、内部服务调用等其他数据传输职责；

2\.  前端提交请求时，仅传递需设置/修改的字段，无需操作的字段主动省略，符合RESTful API最佳实践；

3\.  JSON解析框架反序列化时，请求体中未出现的DTO字段会自动赋值为NULL；

4\.  DTO中NULL属性的核心意义：仅代表“前端未传递该字段”，无任何“主动将该字段设为NULL”的业务意图，需通过合理策略避免误操作；

5\.  数据库表字段配置：以description字段为例，其配置为`text DEFAULT \&\#39;\&\#39;::text NOT NULL`，即该字段默认值为空字符串且不可为NULL，由数据库层面强制保障默认值，与Entity转换策略无关。

# 二、create场景：NULL属性的意义及处理逻辑

create场景核心是新增业务数据，流转链路为：RESTful HTTP创建请求→DTO反序列化→MapStruct Plus转换→ORM操作→数据库插入。本场景下，NULL属性的意义为“前端未设置该字段”，且不同处理策略对最终数据无影响，数据库默认值由表字段配置强制保障。

## 2\.1 第一步：RESTful HTTP创建请求（前端→后端）

前端发起创建请求（如POST /api/tenant）时，仅传递必需字段及主动设置的可选字段，未设置的可选字段不包含在请求体中。此时，未出现的字段对应DTO中的NULL属性，意义为“用户未设置该字段”，其最终值由数据库表字段默认值决定，而非用户主动设为NULL。

示例（创建租户请求）：前端传递type、name、domain，未传递description，请求体如下：

```json
{
  "type": "ENTERPRISE",
  "name": "测试租户",
  "domain": "test.tenant.com"
}
```

## 2\.2 第二步：DTO反序列化（后端接收请求）

后端通过Jackson框架将请求体反序列化为创建DTO，未出现的description字段自动赋值为NULL。此时DTO中的NULL属性，仅表示“前端未传递该字段”，无任何主动设为NULL的意图。

反序列化后DTO状态：

```java
// TenantCreateDTO 反序列化结果
TenantCreateDTO tenantCreateDTO = new TenantCreateDTO();
tenantCreateDTO.setType("ENTERPRISE"); // 前端传递，非NULL
tenantCreateDTO.setName("测试租户");   // 前端传递，非NULL
tenantCreateDTO.setDomain("test.tenant.com"); // 前端传递，非NULL
tenantCreateDTO.setDescription(null); // 前端未传递，反序列化为NULL
```

## 2\.3 第三步：MapStruct Plus转换（DTO→Entity）

该环节核心是将DTO转换为Entity，为ORM操作提供支撑。本场景下，SET\_TO\_NULL与IGNORE两种处理策略对Entity最终状态无差异：

SET\_TO\_NULL策略：DTO中NULL属性（description）会主动赋值给Entity，导致Entity的description字段为NULL；

IGNORE策略（本项目配置）：忽略DTO中NULL属性字段，Entity的description字段保持自身默认值（NULL）；

两种策略转换后，Entity的description字段均为NULL，即create场景下两种策略的Entity结果完全一致，差异不体现在此环节。

转换后的Entity状态（两种策略一致）：

```java
// Tenant 实体转换结果（两种策略一致）
Tenant tenant = new Tenant();
tenant.setType("ENTERPRISE"); // 继承DTO的非NULL值
tenant.setName("测试租户");   // 继承DTO的非NULL值
tenant.setDomain("test.tenant.com"); // 继承DTO的非NULL值
tenant.setDescription(null); // 两种策略均为NULL
```

## 2\.4 第四步：ORM操作（Entity→数据库）

ORM框架根据Entity字段值生成INSERT语句，两种策略下，Entity的description字段均为NULL，ORM会将该字段写入INSERT语句（SET\_TO\_NULL策略）或因NULL值忽略（IGNORE策略），但最终不影响数据库插入结果。

生成的INSERT语句（两种策略对应）：

```sql
-- SET_TO_NULL策略：写入NULL值
INSERT INTO tenant (type, name, domain, description) VALUES ('ENTERPRISE', '测试租户', 'test.tenant.com', null);
-- IGNORE策略：忽略NULL值，不写入该字段
INSERT INTO tenant (type, name, domain) VALUES ('ENTERPRISE', '测试租户', 'test.tenant.com');
```

## 2\.5 第五步：数据库插入（最终持久化）

数据库表中description字段配置为`text DEFAULT \&\#39;\&\#39;::text NOT NULL`，无论ORM是否传递NULL值，数据库都会强制将NULL转换为预设默认值（空字符串），确保字段非NULL且符合默认配置。

两种策略最终插入结果一致：description字段均为数据库默认空字符串，无差异。因此，create场景下，NULL属性的意义为“未设置”，两种处理策略无实际区别，数据库默认值由表字段配置强制保障。

create场景小结：本场景中NULL属性仅表示“前端未设置字段”，处理策略不影响最终数据，数据库表字段的DEFAULT和NOT NULL配置是保障默认值的核心，IGNORE策略的价值在此场景下不体现。

# 三、update场景：NULL属性的意义及处理策略（核心场景）

update场景核心是修改已有数据、仅更新主动修改字段，流转链路为：RESTful HTTP更新请求→DTO反序列化→MapStruct Plus转换→ORM操作→数据库更新。本场景下，NULL属性的意义为“不修改该字段，保留原有值”，IGNORE策略是保障数据安全、避免业务异常的核心选择，也是本项目配置该策略的核心原因。

## 3\.1 第一步：RESTful HTTP更新请求（前端→后端）

前端发起更新请求（如PUT /api/tenant/\{id\}）时，仅传递需修改字段，无需修改字段主动省略。此时，未出现字段对应DTO中的NULL属性，意义为“不修改该字段，保留原有值”，与create场景的“未设置”有本质区别。

示例（更新租户请求）：现有租户数据为type=ENTERPRISE、name=测试租户等，用户仅修改name，请求体如下：

```json
{
  "name": "修改后的测试租户"
}
```

## 3\.2 第二步：DTO反序列化（后端接收请求）

请求体反序列化为更新DTO后，未出现的type、domain、description字段均为NULL。此时DTO中的所有NULL属性，均表示“不修改该字段，保留原有数据”。

反序列化后DTO状态：

```java
// TenantUpdateDTO 反序列化结果
TenantUpdateDTO tenantUpdateDTO = new TenantUpdateDTO();
tenantUpdateDTO.setType(null);        // 前端未传递，反序列化为NULL
tenantUpdateDTO.setName("修改后的测试租户"); // 前端传递，非NULL
tenantUpdateDTO.setDomain(null);      // 前端未传递，反序列化为NULL
tenantUpdateDTO.setDescription(null); // 前端未传递，反序列化为NULL
```

## 3\.3 第三步：MapStruct Plus转换（DTO→Entity，基于原有Entity）

先通过ORM查询原有Entity，再将DTO字段映射至原有Entity，核心是“增量覆盖”——仅用DTO非NULL字段覆盖原有字段，这也是处理NULL属性的关键环节。

结合本项目代码：

```java
// 1. 查询原有租户实体
Tenant existingTenant = tenantRepository.findById(id).orElseThrow(...);
// 2. 将DTO转换到原有Entity中
Tenant update = converter.convert(tenant, existingTenant);
```

SET\_TO\_NULL策略（不当策略）：会用DTO中NULL属性覆盖原有Entity字段，导致原有非NULL字段（如description，数据库存储为默认空字符串）被设为NULL；

IGNORE策略（本项目配置）：忽略DTO中的NULL属性字段，仅用DTO非NULL字段（name）覆盖原有Entity字段，其他字段保持原有值（与数据库存储一致），符合业务逻辑。

两种策略转换后的Entity状态对比：

```java
// existingTenant 原有状态（与数据库一致）
existingTenant.setType("ENTERPRISE");
existingTenant.setName("测试租户");
existingTenant.setDomain("test.tenant.com");
existingTenant.setDescription(""); // 数据库默认空字符串

// SET_TO_NULL策略：被DTO NULL属性覆盖，出现异常
update.setType(null);
update.setName("修改后的测试租户");
update.setDomain(null);
update.setDescription(null);

// IGNORE策略：仅覆盖非NULL字段，状态正常
update.setType("ENTERPRISE"); // 保持原有值
update.setName("修改后的测试租户"); // 被DTO非NULL值覆盖
update.setDomain("test.tenant.com"); // 保持原有值
update.setDescription(""); // 保持原有值（与数据库一致）
```

## 3\.4 第四步：ORM操作（Entity→数据库）

ORM框架生成UPDATE语句，两种策略对NULL属性的处理差异显著：

SET\_TO\_NULL策略：Entity含多个NULL值，ORM生成含这些字段的UPDATE语句，将原有值改为NULL；

IGNORE策略：仅拼接DTO非NULL字段，生成仅修改name的UPDATE语句，示例如下：

```sql
UPDATE tenant SET name = '修改后的测试租户' WHERE id = ?
```

需注意：即使数据库description字段为NOT NULL，若ORM传递NULL值，会触发数据库校验异常，导致更新失败；IGNORE策略可有效避免该问题，正确处理DTO中NULL属性的“不修改”含义。

## 3\.5 第五步：数据库更新（最终持久化）

IGNORE策略下，数据库仅更新name字段的值，其他字段保留原有数据（与数据库配置一致），符合用户“仅修改指定字段”的需求；SET\_TO\_NULL策略会因传递NULL值触发数据库NOT NULL校验异常，或（若未校验）导致原有数据异常。

update场景小结：本场景中NULL属性的意义为“不修改字段”，IGNORE策略可精准识别该含义，避免用DTO NULL属性覆盖原有数据，保障update操作正常，杜绝数据异常，是适配本场景的最优处理策略。

# 四、总结：NULL属性的核心意义与最优处理策略

结合create和update两种核心场景的全链路分析，明确NULL属性的意义及适配的处理策略，核心结论如下：

1. NULL属性的场景化意义：create场景中表示“前端未设置字段”，update场景中表示“不修改字段，保留原有值”，二者本质不同，需通过策略精准区分；

2. create场景处理策略：两种策略无差异，NULL属性的最终值由数据库表字段`DEFAULT \&\#39;\&\#39;::text NOT NULL`强制保障，与转换策略无关；

3. update场景处理策略（核心）：IGNORE策略是最优选择，可忽略DTO中表示“不修改”的NULL属性，避免覆盖原有数据、触发数据库校验异常，贴合RESTful API最佳实践；

4. 策略适配性：IGNORE策略贴合本项目DTO用途及代码实现，可保障数据安全与业务逻辑正确性，非IGNORE策略（如SET\_TO\_NULL）会导致update场景数据异常，与系统设计相悖。

综上，NULL属性的意义具有场景依赖性，MapStruct Plus框架将nullValuePropertyMappingStrategy设为IGNORE，是适配本项目业务场景、保障数据安全的唯一最优处理策略。

# 五、NULL值使用规范

为规范NULL值的使用，避免因NULL值含义混淆、使用不当导致的数据异常和业务逻辑偏差，结合本项目实际场景，制定以下使用规范，适用于数据库设计、业务系统开发及实体类定义全流程：

1\.  数据库字段规范：所有数据库字段需设置为NOT NULL，杜绝字段允许为NULL的情况；对于非必须字段，需明确设置合理的默认值（如字符串类型默认空字符串、数值类型默认0等）；delete time（删除时间）字段是特殊情况，该字段可允许为NULL，因为未删除的数据没有删除时间，但业务中应该用 is delete 标识来判断删除状态。

2\.  业务系统使用规范：业务系统开发中，禁止使用NULL表示具体业务含义，NULL的唯一含义为“没有这个值”，不可用NULL表示“未生效”“已取消”“状态异常”等具体业务状态，需通过明确的枚举值、状态码等方式表达具体业务含义。

3\.  实体类定义规范：DTO、VO、Entity等各类实体类中，必需字段（如业务主键、核心业务字段）不可为NULL，需在代码层面通过注解、校验逻辑等方式强制约束；非必须字段可允许为NULL，但该NULL值仅表示“未赋值”，不可利用该属性的NULL值进行任何业务逻辑判断（如不可通过“if \(entity\.getDescription\(\) == null\)”判断业务状态）。

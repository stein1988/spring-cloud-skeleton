## Findings: No Controller Explicitly Uses "v1" in `@RequestMapping`

**None of the controllers in the project explicitly include "v1" in their `@RequestMapping` path.** Instead, the project uses an **automatic path prefix mechanism** that injects the version number at runtime. Here is the full picture:

---

### 1. The Automatic Prefix Mechanism

The core logic lives in [WebMvcConfiguration.java](file:///d:/Java/spring-cloud-skeleton/module-base/src/main/java/com/lonbon/cloud/base/spring/WebMvcConfiguration.java), specifically in the `extractPrefix` method (lines 101-134). The rules are:

- **If the controller's `@RequestMapping` already contains a version pattern** (matching `/v{digit}` or `/v{digit}.{digit}`), the prefix is `/api/{module}` -- the version in the annotation is preserved as-is.
- **If the controller's `@RequestMapping` does NOT contain a version pattern**, the prefix defaults to `/api/{module}/v1` -- so `v1` is automatically appended.

The module name is extracted from the controller's package using the regex `com.lonbon.cloud.{module}.*`.

### 3. The Version Detection Logic

From [WebMvcConfiguration.java:123](file:///d:/Java/spring-cloud-skeleton/module-base/src/main/java/com/lonbon/cloud/base/spring/WebMvcConfiguration.java#L123):

```java
Pattern versionPattern = Pattern.compile(".*/v(\\d+(\\.\\d+)?).*");
```

This regex matches paths containing `/v1`, `/v2`, `/v1.0`, `/v2.1`, etc. If a match is found, the framework assumes the controller has already specified its version and only prepends `/api/{module}`. If no match is found, it prepends `/api/{module}/v1`.

### 4. Documented Convention

The project's documentation at [07.api路径规范与version版本控制.md](file:///d:/Java/spring-cloud-skeleton/docs/04.Spring%E6%A1%86%E6%9E%B6/07.api%E8%B7%AF%E5%BE%84%E8%A7%84%E8%8C%83%E4%B8%8Eversion%E7%89%88%E6%9C%AC%E6%8E%A7%E5%88%B6.md) confirms:

- The standard URL format is: **`/api/{module}/v{version}/{resource}`**
- The default version is `v1` when no version is specified in the controller's `@RequestMapping`
- Example: A controller with `@RequestMapping("/users")` in the `user` module resolves to `/api/user/v1/users`

### 解析逻辑

按照 URL 格式 `api/{module}/v1/{resource}/{action}`，函数分三步提取权限码：

**1. 提取 module** — 定位 URL 中第一个 `api` 段，取其下一个段作为模块名（如 `user`、`demo`）

**2. 提取 resource** — 定位 URL 中第一个 `v1` 段，取其下一个段作为资源名（如 `roles`、`users`）

**3. 确定 action** — 优先从 URL 路径末尾判断，其次从 HTTP 方法推断：

| 优先级 | 判断条件 | action |
|--------|----------|--------|
| 1 | URL 末段为 `delete` | `delete` |
| 2 | URL 末段为 `update` | `update` |
| 3 | URL 末段为 `add` | `add` |
| 4 | HTTP GET | `query` |
| 5 | HTTP POST | `add` |
| 6 | HTTP PUT/PATCH | `update` |
| 7 | HTTP DELETE | `delete` |

# OpenAPI 配置指南

## 1. 概述

OpenAPI（前身为Swagger）是一种用于描述RESTful API的规范，通过它可以自动生成API文档、客户端代码和测试工具。本项目使用SpringDoc OpenAPI来实现API文档的自动生成。

## 2. 依赖配置

### 2.1 根项目依赖管理

在根项目的`pom.xml`中，通过`dependencyManagement`统一管理SpringDoc的版本：

```xml
<properties>
    <!-- 其他属性... -->
    <springdoc.version>3.0.2</springdoc.version>
</properties>

<dependencyManagement>
    <dependencies>
        <!-- SpringDoc OpenAPI -->
        <dependency>
            <groupId>org.springdoc</groupId>
            <artifactId>springdoc-openapi-starter-webmvc-ui</artifactId>
            <version>${springdoc.version}</version>
        </dependency>
        <!-- 其他依赖... -->
    </dependencies>
</dependencyManagement>
```

### 2.2 模块依赖

在`module-user`模块的`pom.xml`中添加SpringDoc依赖：

```xml
<!-- SpringDoc OpenAPI -->
<dependency>
    <groupId>org.springdoc</groupId>
    <artifactId>springdoc-openapi-starter-webmvc-ui</artifactId>
</dependency>
```

## 3. 配置文件设置

在`module-user/src/main/resources/application.yml`中配置SpringDoc：

```yaml
# SpringDoc 核心配置
springdoc:
  # 1. API文档生成开关（生产环境建议关闭）
  api-docs:
    enabled: true
    path: /v3/api-docs  # OpenAPI规范文件路径（默认）
  # 2. Swagger UI 配置（可视化界面）
  swagger-ui:
    enabled: true
    path: /swagger-ui.html  # UI访问路径（简化版，可直接访问）
    operationsSorter: method  # 接口按HTTP方法排序（GET/POST/PUT/DELETE）
    tagsSorter: alpha         # 标签按字母排序
  # 3. 接口扫描规则（精准扫描，避免冗余）
  packages-to-scan: com.lonbon.cloud.user.application.controller  # 替换为你的控制器包路径
  paths-to-exclude: /error/**, /actuator/**      # 排除无需文档的路径

# OpenAPI 全局元信息（文档的基础说明）
openapi:
  info:
    title: "示例系统API文档"          # 文档标题
    description: "Spring Boot 4.0 + SpringDoc 生成API文档示例" # 文档描述
    version: "v1.0.0"              # 接口版本
    contact: # 维护人信息（可选）
      name: "开发团队"
      email: "dev@example.com"
  servers: # 接口访问环境（可选）
    - url: "http://localhost:8080"
      description: "本地开发环境"
  tags: # 接口分类标签（与控制器@Tag对应）
    - name: "用户接口"
      description: "用户相关的增删改查接口"
```

## 4. 控制器注解使用

在控制器类中使用SpringDoc的注解来描述API：

```java
package com.lonbon.cloud.user.application.controller;

import com.lonbon.cloud.common.utils.Response;
import com.lonbon.cloud.user.domain.dto.TenantCreateDTO;
import com.lonbon.cloud.user.domain.dto.TenantUpdateDTO;
import com.lonbon.cloud.user.domain.entity.Tenant;
import com.lonbon.cloud.user.domain.service.TenantService;
import io.swagger.v3.oas.annotations.Operation;
import io.swagger.v3.oas.annotations.tags.Tag;
import lombok.extern.slf4j.Slf4j;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.validation.annotation.Validated;
import org.springframework.web.bind.annotation.*;
import jakarta.validation.constraints.NotNull;

import java.util.Optional;
import java.util.UUID;

@Slf4j
@RestController
@RequestMapping("/api/tenants")
@Tag(name = "租户", description = "租户操作")
public class TenantController {

    @Autowired
    private TenantService tenantService;

    @PostMapping
    @Operation(summary = "创建", description = "创建description")
    public Response<UUID> create(@RequestBody @Validated @NotNull TenantCreateDTO tenant) {
        Tenant createdTenant = tenantService.createTenant(tenant);
        return Response.success(createdTenant.getId(), "Tenant created successfully");
    }

    @PostMapping("/{id}/delete")
    @Operation(summary = "删除", description = "删除description")
    public Response<UUID> delete(@PathVariable("id") UUID id) {
        tenantService.deleteTenant(id);
        return Response.success(id, "Tenant deleted successfully");
    }

    @PostMapping("/{id}/update")
    @Operation(summary = "更新", description = "更新description")
    public Response<UUID> update(@PathVariable("id") UUID id, @RequestBody @Validated TenantUpdateDTO tenant) {
        tenantService.updateTenant(id, tenant);
        return Response.success(id, "Tenant updated successfully");
    }

    @GetMapping("/{id}")
    @Operation(summary = "获取", description = "获取description")
    public Response<Tenant> getTenantById(@PathVariable("id") UUID id) {
        Optional<Tenant> tenant = tenantService.getTenantById(id);
        return tenant.map(Response::success).orElseGet(() -> Response.error("Tenant not found"));
    }
}
```

## 5. 核心注解说明

| 注解 | 位置 | 作用 | 示例 |
|------|------|------|------|
| `@Tag` | 控制器类 | 标记接口分类 | `@Tag(name = "租户", description = "租户操作")` |
| `@Operation` | 控制器方法 | 标记接口功能 | `@Operation(summary = "创建", description = "创建租户")` |
| `@Parameter` | 方法参数 | 标记参数说明 | `@Parameter(description = "租户ID", required = true)` |
| `@RequestBody` | 方法参数 | 标记请求体 | `@RequestBody @Validated TenantCreateDTO tenant` |
| `@Schema` | 实体类/字段 | 标记实体/字段说明 | `@Schema(description = "租户名称", required = true)` |
| `@ApiResponses` | 方法 | 标记响应码和说明 | `@ApiResponses({@ApiResponse(responseCode = "200", description = "成功"), @ApiResponse(responseCode = "400", description = "参数错误")})` |

## 7. 最佳实践

### 7.1 控制器层面

1. **类级别**：使用`@Tag`注解标记控制器的功能分类，与`openapi.tags`配置对应
2. **方法级别**：使用`@Operation`注解详细描述接口的功能和参数
3. **参数级别**：使用`@Parameter`或`@Schema`注解描述参数的含义和约束

### 7.2 配置层面

1. **生产环境**：建议关闭API文档生成，设置`springdoc.api-docs.enabled: false`和`springdoc.swagger-ui.enabled: false`
2. **精准扫描**：通过`springdoc.packages-to-scan`指定需要生成文档的控制器包
3. **排除路径**：通过`springdoc.paths-to-exclude`排除不需要生成文档的路径（如错误页、监控端点等）

### 7.3 实体层面

1. **DTO类**：为请求和响应的DTO类添加`@Schema`注解，描述字段含义
2. **字段说明**：为重要字段添加`description`、`example`、`required`等属性
3. **验证注解**：结合`jakarta.validation`注解，生成更准确的参数验证信息

## 8. 版本兼容性

- **Spring Boot 4.0**：使用SpringDoc OpenAPI v3.0.0+（本项目使用3.0.2）
- **Spring Boot 3.x**：使用SpringDoc OpenAPI v2.5.0+
- **Spring Boot 2.x**：使用SpringDoc OpenAPI v1.6.0+

## 9. 常见问题与解决方案

### 9.1 文档不显示控制器

**问题**：Swagger UI中看不到控制器的API

**解决方案**：
- 检查`springdoc.packages-to-scan`配置是否正确
- 确保控制器类使用了`@RestController`或`@Controller`注解
- 确保控制器方法使用了`@GetMapping`、`@PostMapping`等HTTP方法注解

### 9.2 实体属性不显示

**问题**：API文档中请求/响应的实体属性不显示

**解决方案**：
- 为实体类添加`@Schema`注解
- 确保实体类使用了`@Data`或提供了getter方法
- 检查实体类是否被正确扫描

### 9.3 验证信息不显示

**问题**：API文档中没有显示参数验证信息

**解决方案**：
- 使用`jakarta.validation`注解（如`@NotNull`、`@NotEmpty`等）
- 确保方法参数使用了`@Validated`或`@Valid`注解
- 检查是否添加了`spring-boot-starter-validation`依赖

## 10. 总结

SpringDoc OpenAPI为项目提供了强大的API文档生成能力，通过简单的配置和注解，就可以生成规范、美观的API文档。本项目的配置方案已经考虑了Spring Boot 4.0的兼容性，确保在不同环境下都能正常使用。

通过合理使用SpringDoc的注解和配置，可以：
- 提高API的可维护性和可读性
- 减少文档维护的工作量
- 为前端开发和测试提供清晰的API参考
- 提升整个项目的专业度和可协作性

# HTTP接口Path参数结构化

## 概述

在Spring Boot应用中，处理HTTP GET请求时，传统的做法是在控制器方法中逐个声明参数。这种方式在参数较多时会导致方法签名冗长，可读性差，且难以维护。通过使用结构化的Java类来解析GET参数，可以使代码更加简洁、清晰和可维护。

## 核心概念

### 1. 参数结构化

参数结构化是指将多个相关的请求参数封装到一个Java类中，通过该类来接收和处理请求参数。这种方式有以下优点：

- 减少控制器方法的参数数量，使方法签名更加简洁
- 提高代码的可读性和可维护性
- 便于参数的验证和处理
- 支持复杂的参数结构

### 2. @ParameterObject注解

`@ParameterObject`是Spring Doc提供的注解，用于标记一个类作为参数对象。它的主要作用是：

- 告诉Spring Doc该类是一个参数对象，在生成API文档时会将其属性作为独立的参数显示
- 与Spring MVC的参数绑定机制配合，自动将请求参数映射到对象的属性

## 实现方式

### 1. 创建参数类

创建一个包含所有请求参数的Java类，并添加必要的注解：

```java
@ParameterObject
@Data
public class TenantQueryDTO {

    @EasyWhereCondition(type = EasyWhereCondition.Condition.EQUAL)
    @Schema(description = "名称，精确查询")
    private String name;

    @Schema(description = "名称，模糊查询")
    @EasyWhereCondition(propName = "name", type = EasyWhereCondition.Condition.LIKE)
    private String nameLike;

    private String description;

    private String domain;
}
```

### 2. 创建分页参数类

创建一个专门的分页参数类，用于处理分页和排序：

```java
@Data
@ParameterObject
@Schema(description = "自定义分页参数")
public class Pageable {

    @Schema(description = "页码（从0开始）", example = "0", defaultValue = "0")
    private Integer page;

    @Schema(description = "每页条数", example = "10", defaultValue = "10")
    private Integer size;

    @Schema(description = "排序规则，格式：字段名.方向（asc/desc），可以接受多个参数，注意分隔符是点，不是逗号",
            example = "createdAt.desc")
    private List<String> sort;

    @Schema(description = "解析后的排序数据，不显示在接口文档中", hidden = true)
    @Setter(AccessLevel.NONE)   // 内部缓存字段，禁用Setter
    private List<Sortable> sortables;

    // 其他方法...
}
```

### 3. 在控制器中使用

在控制器方法中直接使用参数类作为参数：

```java
@GetMapping()
@Operation(summary = "查询", description = "查询description")
public Response<PageResult<Tenant>> getTenants(
        TenantQueryDTO query,
        Pageable pageable
) {
    log.info("query: {}, pageable: {}", query, pageable);
    PageResult<Tenant> tenants = tenantService.getTenants(query, pageable);
    return Response.success(tenants);
}
```

## 注解详解

### 1. @ParameterObject

- **作用**：标记一个类作为参数对象，Spring Doc会将其属性作为独立的参数显示在API文档中
- **使用场景**：当一个请求需要多个参数时，将这些参数封装到一个类中
- **注意事项**：该注解是Spring Doc提供的，需要添加相应的依赖

### 2. @Schema

- **作用**：用于描述参数的元数据，如描述、示例值、默认值等
- **使用场景**：为API文档提供更详细的参数信息
- **属性**：
  - `description`：参数描述
  - `example`：示例值
  - `defaultValue`：默认值
  - `hidden`：是否在API文档中隐藏该参数

## 请求示例

### 传统方式

传统方式需要在方法中逐个声明参数：

```java
@GetMapping()
public Response<PageResult<Tenant>> getTenants(
        @RequestParam("name") String name,
        @RequestParam("nameLike") String nameLike,
        @RequestParam("description") String description,
        @RequestParam("domain") String domain,
        @RequestParam("page") Integer page,
        @RequestParam("size") Integer size,
        @RequestParam("sort") List<String> sort
) {
    // 处理逻辑
}
```

### 结构化方式

结构化方式使用参数类：

```java
@GetMapping()
public Response<PageResult<Tenant>> getTenants(
        TenantQueryDTO query,
        Pageable pageable
) {
    // 处理逻辑
}
```

## 优势对比

| 特性 | 传统方式 | 结构化方式 |
|------|---------|------------|
| 代码简洁性 | 差，参数多时方法签名冗长 | 好，方法签名简洁 |
| 可维护性 | 差，参数分散在方法签名中 | 好，参数集中在一个类中 |
| 可读性 | 差，参数过多时难以理解 | 好，参数结构清晰 |
| 扩展性 | 差，添加新参数需要修改方法签名 | 好，只需修改参数类 |
| 验证支持 | 可以，但需要在方法参数上添加注解 | 好，可以在参数类中集中添加验证注解 |
| API文档生成 | 可以，但需要在方法参数上添加注解 | 好，参数类中的注解会自动反映到API文档中 |

## 最佳实践

1. **参数分类**：将相关的参数组织到一个参数类中，如查询参数、分页参数等

2. **合理命名**：参数类的名称应该能够清晰地表达其用途，如`TenantQueryDTO`、`Pageable`等

3. **添加注解**：为参数类和其属性添加适当的注解，如`@ParameterObject`、`@Schema`等，以生成清晰的API文档

4. **默认值处理**：在参数类中处理默认值，如页码默认为0，每页条数默认为10等

5. **参数验证**：在参数类中添加验证注解，如`@NotNull`、`@Size`等，以确保参数的合法性

6. **排序参数处理**：对于排序参数，提供清晰的格式说明，并在参数类中进行解析

## 注意事项

**重要提醒**：当在同一个控制器方法中使用多个参数类时，各个参数类的字段名称不能重复。如果不同参数类中存在同名字段，Spring MVC在绑定参数时会产生冲突，导致参数值覆盖或绑定失败。

例如，不要在`TenantQueryDTO`和`Pageable`中都定义名为`name`的字段，这样会导致请求参数`name`的值无法正确绑定到对应的参数类中。

## 常见问题

### Q: 为什么我的参数没有被正确映射？

A: 请检查以下几点：
- 参数类是否添加了`@Data`或相应的getter/setter方法
- 请求参数名是否与参数类的属性名一致
- 是否正确使用了`@ParameterObject`注解

### Q: 如何处理复杂的参数结构？

A: 可以创建嵌套的参数类，将复杂的参数结构分解为多个层次的参数类

### Q: 如何在API文档中显示参数的描述？

A: 使用`@Schema`注解为参数添加描述信息

## 总结

HTTP接口参数结构化是一种优雅的方式，它可以使代码更加简洁、清晰和可维护。通过使用`@ParameterObject`注解和结构化的参数类，可以有效地管理和处理HTTP GET请求的参数，提高开发效率和代码质量。

在实际项目中，建议采用这种方式来处理复杂的请求参数，特别是当参数数量较多时。这样不仅可以使代码更加整洁，还可以生成更加清晰的API文档，提高API的可理解性和可使用性。

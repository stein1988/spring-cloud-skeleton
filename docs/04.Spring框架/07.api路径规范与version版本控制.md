# API 路径规范与版本控制

## 1. 概述

在微服务架构中，API 路径规范和版本控制是非常重要的设计要素。良好的 API 路径规范可以提高 API 的可读性和可维护性，而版本控制则可以确保 API 变更的平滑过渡，避免破坏现有客户端。

本文档将介绍本项目采用的 API 路径规范与版本控制方案，以及它如何与 springdoc 无缝配合，生成正确的接口文档。

## 2. 实现方案

### 2.1 核心实现

本项目采用了基于 Spring WebMvc 配置的自动路径前缀添加方案，通过 `WebMvcConfiguration` 类实现：

### 2.2 路径前缀生成规则

1. **模块名提取**：从包名中提取模块名，匹配正则表达式 `com.lonbon.cloud.{module}.*`
2. **版本号检测**：检查控制器的 `@RequestMapping` 注解是否包含版本号（如 `/v1`）
3. **前缀生成**：
   - 如果控制器的 `@RequestMapping` 中已包含版本号（如 `/v2/users`），则前缀为 `/api/{module}`
   - 如果控制器的 `@RequestMapping` 中未包含版本号（如 `/users`），则前缀为 `/api/{module}/v1`

### 2.3 示例

#### 示例 1：控制器未指定版本号

```java
// 包名：com.lonbon.cloud.user.application.controller
@RestController
@RequestMapping("/users")
public class UserController {
    // ...
}
```

生成的完整路径：`/api/user/v1/users`

#### 示例 2：控制器已指定版本号

```java
// 包名：com.lonbon.cloud.order.application.controller
@RestController
@RequestMapping("/v2/orders")
public class OrderController {
    // ...
}
```

生成的完整路径：`/api/order/v2/orders`

## 3. 与 springdoc 的无缝配合

### 3.1 优势

本方案与 springdoc 无缝配合的优势在于：

1. **自动文档生成**：springdoc 会自动检测控制器的 `@RequestMapping` 注解，并结合我们配置的路径前缀，生成正确的 API 文档
2. **版本号显示**：在文档中清晰显示 API 的版本号，便于客户端了解和使用
3. **模块划分**：通过路径前缀中的模块名，在文档中自动对 API 进行分类，提高文档的可读性
4. **无侵入性**：不需要在控制器上添加额外的注解或配置，保持代码的简洁性

### 3.2 实现原理

springdoc 在生成 API 文档时，会：
1. 扫描所有带有 `@RestController` 或 `@Controller` 注解的类
2. 读取这些类的 `@RequestMapping` 注解，获取路径信息
3. 结合 Spring MVC 的路径匹配配置，生成完整的 API 路径
4. 将这些信息组织成 Swagger 文档

由于我们的方案是通过 Spring MVC 的 `PathMatchConfigurer` 配置路径前缀，而不是修改控制器的 `@RequestMapping` 注解，因此 springdoc 能够正确地检测和处理这些路径前缀，生成准确的 API 文档。

## 4. Spring Boot 4 原生 API 版本控制方案

Spring Boot 4 引入了原生的 API 版本控制支持，通过 `@ApiVersion` 注解或配置文件指定 API 版本。但是截止到2026.04当前版本，这种方式在与 springdoc 配合时还存在问题，springdoc 可能无法正确识别和处理 Spring Boot 4 的原生 API 版本控制机制，导致生成的文档中版本号显示不正确或缺失

## 5. 最佳实践

### 5.1 控制器命名和包结构

为了确保路径前缀生成的准确性，建议遵循以下命名规范：

1. **包结构**：使用 `com.lonbon.cloud.{module}.application.controller` 的包结构，其中 `{module}` 为模块名
2. **控制器命名**：控制器类名建议使用 `{Resource}Controller` 的形式，如 `UserController`、`OrderController`
3. **路径命名**：控制器的 `@RequestMapping` 路径建议使用小写字母和连字符，如 `/users`、`/user-details`

### 5.2 版本控制策略

1. **版本号格式**：建议使用 `v{major}` 或 `v{major}.{minor}` 的格式，如 `v1`、`v1.0`
2. **版本升级**：当 API 发生不兼容变更时，应增加主版本号；当 API 仅添加新功能且保持向后兼容时，可增加次版本号
3. **路径一致性**：同一模块的 API 应使用相同的版本号格式，保持路径的一致性

## 6. 总结

本项目采用的 API 路径规范与版本控制方案，通过基于 Spring WebMvc 配置的自动路径前缀添加机制，实现了以下目标：

1. **统一的 API 路径规范**：所有 API 路径都遵循 `/api/{module}/v{version}/{resource}` 的格式
2. **灵活的版本控制**：支持在控制器级别指定版本号，或使用默认版本号
3. **与 springdoc 的无缝集成**：自动生成正确的 API 文档，无需额外配置
4. **低侵入性**：不需要修改现有控制器代码，只需添加一个配置类
5. **易于维护**：集中管理路径前缀生成规则，便于后续调整和扩展

相比 Spring Boot 4 的原生 API 版本控制，本方案在与 springdoc 的配合方面表现更好，能够生成更加准确和完整的 API 文档，同时保持了代码的简洁性和灵活性。

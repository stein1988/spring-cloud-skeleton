# 多租户养老系统 最终设计方案（含演进过程、选择理由、数据表结构）

# 一、设计演进过程（全流程梳理）

本方案基于多租户架构，从基础的人员与权限体系设计出发，逐步适配养老系统专属场景（老人管理、家属登录），解决数据隔离、权限管控、系统对接、性能优化等核心问题，演进过程分为9个关键阶段，每个阶段均围绕业务需求和架构痛点展开优化。

## 1\.1 阶段一：核心实体定义（解决User命名歧义）

### 演进背景

传统架构中“User”表存在语义歧义，无法区分“自然人主体”“登录账号”“组织任职身份”，导致后续多租户、多角色管理混乱，难以适配养老系统中“老人（非登录）、员工（登录）、家属（登录）”的多角色场景。

### 解决方案

拆分原User表为三个核心实体，明确语义边界：

- Person：全局唯一的自然人顶层抽象，存储与登录、组织、业务无关的基础信息，跨租户复用，覆盖所有自然人（老人、员工、家属、访客）。

- Staff：租户内的员工任职身份，关联Person，存储部门、岗位等组织相关信息，仅用于内部员工场景。

- Account：系统登录凭证与权限挂载主体，关联Person（必关联）、Staff（内部员工非空），按租户隔离，支持多账号关联同一Person。

## 1\.2 阶段二：核心表结构与RBAC权限体系设计（基础架构落地）

### 演进背景

明确核心实体后，需落地基础表结构，同时搭建企业级权限管控体系，满足养老系统中“管理员、员工、家属”的不同权限需求，确保数据安全。

### 解决方案

设计核心业务表（Person/Staff/Account）及6张RBAC权限表，明确权限执行链路，实现“账号\-角色\-权限”的分级管控，为后续多租户扩展奠定基础。

## 1\.3 阶段三：多租户架构改造（适配连锁养老场景）

### 演进背景

养老系统多为连锁化运营（集团\-分公司\-养老院/社区），需要实现不同院区的数据隔离，同时支持跨院区管理，传统单租户架构无法满足需求。

### 解决方案

采用“共享数据库\+共享Schema\+tenant\_id字段隔离”方案（企业级首选），新增租户主表t\_tenant，对Staff、Account及RBAC表添加tenant\_id字段，通过MyBatis\-Plus租户插件实现数据隔离，Person表因全局唯一，不加tenant\_id，支持跨租户复用。

## 1\.4 阶段四：登录体系扩展（适配多角色登录）

### 演进背景

养老系统需要支持员工（账号密码）、家属（手机号/微信）登录，同时需适配多租户场景，确保登录后仅能访问对应租户数据，原有登录体系无法满足多方式、多租户登录需求。

### 解决方案

扩展Account表支持手机号登录，新增t\_account\_third（第三方账号关联表）支持微信授权登录，新增t\_sms\_code（短信验证码表）支持手机号验证码登录，优化登录流程，适配多租户选择，实现微信unionid关联全局Person，跨租户复用登录身份。

## 1\.5 阶段五：Person表兼容优化（解决信息填写痛点）

### 演进背景

创建Account时（如家属、临时访客），用户不想填写完整人员信息，原有Person表要求实名填写，无法适配“极简填写、延迟实名”场景，影响操作效率。

### 解决方案

改造Person表，新增person\_type字段，支持三种模式：实名自然人（person\_type=1）、极简人员（person\_type=2）、虚拟/系统账号（person\_type=3），实现“自动创建、延迟实名、极简填写”，用户创建Account时无需手动操作Person表，后台自动关联/创建Person，提升操作效率。

## 1\.6 阶段六：外部关联字段设计（支持老系统迁移与多系统对接）

### 演进背景

养老系统需对接老养老系统、医保系统、医院系统、门禁系统，同时需要完成老系统数据迁移，原有表结构无外部关联字段，无法实现数据双向同步和历史追溯。

### 解决方案

采用“业务表主外部关联字段\+通用外部关联明细表”双轨设计：在Person/Account/Staff表新增external\_id、external\_system等字段，用于核心系统对接；新建t\_external\_reference表，支持N个外部系统对接，实现老系统数据迁移、多系统双向同步，保留历史数据快照，满足对账需求。

## 1\.7 阶段七：Person表命名优化（规范化）

### 演进背景

原Person表命名虽能表达“自然人”语义，但在企业级架构中不够规范，且存在被泛化使用的风险，需优化表名，确保语义精准、符合规范化要求。

### 解决方案

推荐将Person表替换为t\_individual（DDD国际通用，语义精准，无歧义），备选方案包括t\_natural\_person（强合规场景）、t\_person\_base（存量优化场景），明确禁用t\_user、t\_member等歧义命名，同时制定全链路命名规范，确保数据库、代码、文档术语统一。

## 1\.8 阶段八：租户层级实现（适配集团化管理）

### 演进背景

连锁养老集团需要实现“集团\-分公司\-养老院”的树形层级管理，上级租户需查看/管理下级租户数据，原有单级多租户架构无法实现层级管控和数据权限继承。

### 解决方案

改造t\_tenant表，新增parent\_tenant\_id、tenant\_level、tenant\_path等树形字段，实现无限层级租户（建议控制在5层内）；通过Path前缀查询、MySQL递归CTE、MyBatis\-Plus树形插件实现树形查询；扩展数据权限插件，实现上级租户数据权限继承，下级租户仅能访问自身数据。

## 1\.9 阶段九：上级租户管理员登录管理下级租户（集团化运营需求）

### 演进背景

集团管理员需要直接登录下级养老院租户进行管理、巡检、故障排查，原有架构仅支持单租户登录，无法实现跨租户无密码切换，且存在安全合规风险。

### 解决方案

设计两种切换模式（权限继承模式、身份代理模式），新增跨租户切换权限和t\_tenant\_switch\_log审计表（全链路留痕），扩展会话上下文和JWT Token，实现上级→下级无密码切换，严格控制权限边界，满足等保合规要求，全程不触碰下级账号密码。

## 1\.10 阶段十：养老系统专属适配（老人、家属管理）

### 演进背景

养老系统核心服务对象为老人，老人无需登录系统，但需记录详细业务信息（护理等级、病史等），同时家属需要登录系统查看老人信息，需适配“老人非登录、家属登录”的专属场景。

### 解决方案

将老人纳入t\_individual体系（无需关联Account），新增t\_elder表（养老专属业务子表，与t\_individual 1:1关联），存储老人护理、健康等业务信息；新增t\_elder\_family表，关联老人、家属（t\_individual）和家属登录账号（Account），复用原有登录和权限体系，实现家属登录查看专属老人信息。

## 1\.11 阶段十一：Person表性能优化（解决数据量担忧）

### 演进背景

担心t\_individual表承载老人、员工、家属等所有自然人数据后，数据量过大导致性能瓶颈，影响查询效率。

### 解决方案

通过“索引优化\+冷热数据分离\+分库分表预案\+Redis缓存”四层优化，确保t\_individual表可轻松承载千万级数据：优化索引设计覆盖所有高频查询；定期将不活跃数据归档到历史表；提前规划分库分表方案（未来数据量过亿时使用）；缓存热点数据，减少数据库压力。

# 二、方案选择理由（核心决策依据）

所有设计方案均围绕“业务适配、架构兼容、性能可控、未来扩展、合规安全”五大核心原则，结合养老系统连锁化、多角色、多系统对接的业务特点，每个方案的选择均有明确的决策依据，具体如下：

## 2\.1 核心实体拆分（Person/Staff/Account）选择理由

- 解决语义歧义：彻底解决传统User表“自然人、登录账号、组织身份”混淆的问题，明确各实体职责，降低后续维护成本。

- 适配多角色场景：完美覆盖养老系统“老人（非登录）、员工（登录）、家属（登录）、访客”等多角色，避免角色混淆导致的权限漏洞。

- 架构可扩展：各实体职责单一，便于后续扩展（如新增访客表、外包表），无需重构核心架构。

## 2\.2 多租户方案（共享数据库\+tenant\_id隔离）选择理由

- 成本最优：无需部署多个数据库，降低服务器、运维成本，适合连锁养老机构的成本控制需求。

- 隔离度适中：tenant\_id字段隔离可实现租户间数据完全隔离，同时支持上级租户数据权限继承，适配集团化管理。

- 兼容原有架构：仅需新增字段，不修改原有表结构和业务逻辑，改造难度低，可平滑过渡。

## 2\.3 登录体系扩展选择理由

- 多方式适配：覆盖账号密码、手机号验证码、微信授权三种登录方式，满足员工、家属的不同登录习惯（家属更偏好微信、手机号快速登录）。

- 多租户兼容：登录时支持租户选择或域名自动匹配，确保登录后仅能访问对应租户数据，符合数据隔离要求。

- 复用原有体系：无需新增登录框架，复用RBAC权限体系，降低开发和维护成本。

## 2\.4 Person表兼容优化（三种模式）选择理由

- 提升操作效率：用户创建Account时无需填写完整人员信息，后台自动创建Person，减少操作步骤，适配家属、临时访客的快速注册需求。

- 合规适配：支持实名模式（老人、员工必填身份证），符合养老行业实名登记要求；同时支持极简、虚拟模式，适配特殊场景（如流浪老人）。

- 平滑过渡：后续可补全人员信息，无需修改关联表，不影响业务连续性。

## 2\.5 外部关联双轨设计选择理由

- 灵活扩展：主外部关联字段保证核心系统对接的性能，通用明细表支持N个外部系统对接，无需修改业务表结构，适配养老系统多外部系统对接需求（医保、医院、门禁）。

- 历史追溯：保留老系统原始数据快照，便于数据迁移对账，符合养老行业数据可追溯要求。

- 兼容原有设计：复用原有表结构，不影响现有业务，改造无侵入。

## 2\.6 Person表命名优化（t\_individual）选择理由

- 语义精准：DDD国际通用命名，明确“自然人个体”语义，与Account、Staff完全解耦，无歧义，避免泛化使用。

- 规范化：符合企业级架构命名规范，便于团队协作和后续维护，降低沟通成本。

- 平滑迁移：可通过双写过渡、数据同步，实现从t\_person到t\_individual的平滑迁移，不影响业务运行。

## 2\.7 租户层级（树形租户）选择理由

- 适配集团化运营：完美支持“集团\-分公司\-养老院”的层级管理，满足上级租户对下级租户的统一管理需求。

- 性能可控：通过tenant\_path前缀索引、递归查询优化，避免层级过深导致的性能问题，同时控制层级深度（建议5层内）。

- 权限继承：支持数据权限继承，上级租户可查看下级数据，下级租户仅能访问自身数据，符合权限管控要求。

## 2\.8 上级租户切换模式选择理由

- 安全合规：全程不获取、不暴露下级账号密码，全链路审计留痕，满足等保合规要求，避免安全风险。

- 场景适配：两种模式分别适配“集团统一管理”和“故障排查”场景，兼顾效率和安全性。

- 架构兼容：复用原有会话、权限体系，无需重构核心代码，改造难度低。

## 2\.9 养老专属表（t\_elder/t\_elder\_family）选择理由

- 职责单一：t\_individual存储基础信息，t\_elder存储养老专属业务信息，t\_elder\_family关联家属，避免表结构臃肿，便于维护。

- 扩展灵活：老人无需登录（不关联Account），家属需要登录时仅需关联Account，复用原有登录体系，未来可扩展老人临时登录权限。

- 数据一致：老人关联t\_individual，避免基础信息重复录入，确保数据一致性，同时便于跨系统对接。

## 2\.10 Person表性能优化选择理由

- 成本可控：优先采用索引优化、冷热数据分离等低成本方案，分库分表作为预案，避免过度设计。

- 性能保障：覆盖高频查询场景，缓存热点数据，确保千万级数据毫秒级查询，满足养老系统日常操作需求。

- 未来兼容：分库分表预案可应对未来数据量增长，无需重构核心架构，降低扩展成本。

# 三、最终数据表结构（完整规范，可直接落地）

所有数据表均遵循“蛇形命名、前缀t\_、职责单一、多租户隔离”原则，包含核心业务表、RBAC权限表、外部关联表、养老专属表、审计表，字段注释清晰，索引设计优化，关联关系明确，具体如下：

## 3\.1 全局自然人表（t\_individual，原t\_person）

```sql
CREATE TABLE t_individual (
    individual_id      BIGINT PRIMARY KEY COMMENT '自然人主体ID（全局唯一）',
    real_name          VARCHAR(50) NULL COMMENT '真实姓名',
    id_card            VARCHAR(18) NULL COMMENT '身份证号（实名模式必填）',
    gender             TINYINT COMMENT '性别（1=男 2=女 0=未知）',
    birth_date         DATE COMMENT '出生日期',
    phone              VARCHAR(20) COMMENT '联系手机号',
    email              VARCHAR(100) COMMENT '邮箱',
    avatar             VARCHAR(255) COMMENT '头像',
    person_type        TINYINT DEFAULT 2 COMMENT '主体类型 1=实名自然人 2=极简人员 3=虚拟/系统账号',
    wx_unionid         VARCHAR(100) COMMENT '微信全局唯一标识（用于微信登录关联）',
    remark             VARCHAR(500) COMMENT '备注',
    -- 外部关联字段（复用外部对接设计）
    external_id        VARCHAR(100) COMMENT '主外部系统ID',
    external_system    VARCHAR(50) COMMENT '主外部系统标识（OLD_SYS/HR_SYS/DOOR_SYS等）',
    external_create_time DATETIME COMMENT '外部系统原始创建时间',
    external_update_time DATETIME COMMENT '外部系统原始更新时间',
    external_snapshot  JSON COMMENT '外部系统原始数据快照（JSON格式，用于对账）',
    -- 基础字段
    create_time        DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time        DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    -- 索引设计（优化高频查询）
    UNIQUE KEY uk_id_card (id_card) COMMENT '身份证唯一索引（实名校验）',
    UNIQUE KEY uk_phone (phone) COMMENT '手机号唯一索引（登录/关联）',
    UNIQUE KEY uk_unionid (wx_unionid) COMMENT '微信unionid唯一索引（跨租户关联）',
    UNIQUE KEY uk_tenant_external (tenant_id, external_system, external_id) COMMENT '租户+外部系统+外部ID唯一（外部对接）',
    INDEX idx_person_type (person_type) COMMENT '人员类型索引（筛选场景）',
    INDEX idx_create_time (create_time) COMMENT '创建时间索引（统计场景）'
);
```

## 3\.2 自然人历史表（t\_individual\_history，冷热数据分离用）

```sql
-- 结构与t_individual完全一致，用于归档不活跃数据
CREATE TABLE t_individual_history LIKE t_individual;
-- 新增归档时间字段
ALTER TABLE t_individual_history
ADD COLUMN archive_time DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '归档时间';
```

## 3\.3 员工任职表（t\_staff）

```sql
CREATE TABLE t_staff (
    staff_id           BIGINT PRIMARY KEY COMMENT '员工任职ID',
    individual_id      BIGINT NOT NULL COMMENT '关联全局自然人ID（1:1）',
    tenant_id          BIGINT NOT NULL COMMENT '所属租户ID（多租户隔离）',
    dept_id            BIGINT COMMENT '关联部门ID',
    post_id            BIGINT COMMENT '关联岗位ID',
    job_no             VARCHAR(50) COMMENT '工号（租户内唯一）',
    entry_date         DATE COMMENT '入职日期',
    leave_date         DATE COMMENT '离职日期',
    staff_status       TINYINT DEFAULT 1 COMMENT '状态 1=在职 0=离职',
    leader_id          BIGINT COMMENT '直属领导ID（关联自身）',
    work_type          TINYINT COMMENT '用工类型 1=正式 2=外包 3=临时',
    remark             VARCHAR(500) COMMENT '备注',
    create_time        DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time        DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    -- 索引
    UNIQUE KEY uk_tenant_jobno (tenant_id, job_no) COMMENT '租户内工号唯一',
    INDEX idx_individual_id (individual_id) COMMENT '关联自然人索引',
    INDEX idx_tenant_status (tenant_id, staff_status) COMMENT '租户+员工状态索引'
);
```

## 3\.4 登录账号表（t\_account）

```sql
CREATE TABLE t_account (
    account_id         BIGINT PRIMARY KEY COMMENT '登录账号ID',
    individual_id      BIGINT NOT NULL COMMENT '关联全局自然人ID（1:1）',
    staff_id           BIGINT COMMENT '关联员工ID（内部员工非空，家属/访客为空）',
    tenant_id          BIGINT NOT NULL COMMENT '所属租户ID（多租户隔离）',
    username           VARCHAR(50) NOT NULL COMMENT '登录账号（租户内唯一）',
    password           VARCHAR(100) NOT NULL COMMENT '加密密码（MD5/SHA256）',
    salt               VARCHAR(50) COMMENT '密码盐值',
    phone              VARCHAR(20) COMMENT '手机号（可用于登录，租户内唯一）',
    account_status     TINYINT DEFAULT 1 COMMENT '账号状态 1=正常 0=禁用 2=锁定',
    last_login_time    DATETIME COMMENT '最后登录时间',
    last_login_ip      VARCHAR(50) COMMENT '最后登录IP',
    create_time        DATETIME DEFAULT CURRENT_TIMESTAMP,
```

# 四、核心关联关系总结

- t\_individual（1:N）→ t\_staff：一个自然人可在多个租户担任多个员工角色。

- t\_individual（1:N）→ t\_account：一个自然人可拥有多个登录账号（不同租户/不同场景）。

- t\_staff（1:1）→ t\_account：一个员工任职记录绑定一个登录账号。

- t\_individual（1:1）→ t\_elder：一个自然人对应一条老人业务记录（老人无需关联t\_account）。

- t\_elder（1:N）→ t\_elder\_family：一个老人可对应多个家属。

- t\_elder\_family（1:1）→ t\_individual：一个家属对应一个自然人，可关联t\_account（支持登录）。

- t\_tenant（1:N）→ 所有业务表（除t\_individual、t\_individual\_history）：租户隔离，树形层级关联。

- t\_account（N:M）→ t\_role（通过t\_account\_role）：一个账号对应多个角色。

- t\_role（N:M）→ t\_permission（通过t\_role\_permission）：一个角色对应多个权限。
